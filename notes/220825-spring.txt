// 어떤 기술이든 핵심 컨셉, 뼈대, 줄기, 알맹이는 항상 단순하다.
그 단순한 컨셉이 충분히 공감을 얻는다면 거기에 너도 나도 조금씩 살을 덧대어서 큰 기술이 되는 것.
그렇다면 스프링의 핵심 컨셉은 무엇인가?
SOLID 객체 지향 언어 Java에 가장 어울리는 프로그래밍을 설계할 수 있도록 도와주는 도구.
객체 지향적으로 코드를 짜고, 가장 Java답게, 객체 지향 언어답게 짤 수 있도록 만들어주는 도구.



// 그러다면 객체 지향 프로그래밍이란 무엇인가?
가장 객체 지향 언어답게, 가장 객체 지향 언어에 어울리게 프로그래밍을 설계할 수 있도록 해준다며.
그렇다면 그게 뭔데?
/ 프로그램을 '객체'들의 모임으로 파악하고, 각 객체들이 서로 데이터를 주거나 받는 구조로 설계하는 것.
/ 역할과 개체를 분리해서 생각하기. 운전자 역할이 따로 있고, 운전자 개체가 따로 있고.
운전자 김 씨도 있고, 박 씨도 있고, 최 씨도 있어. 김 씨가 오늘 당직이었는데 급한 일이 생겨서 최 씨가 대신 운전할 수도 있어.
오늘 운전자 개체가 김 씨인지 최 씨인지 알 필요는 없어. 그냥 그 날 그 시간에 운전자 역할을 수행해주기만 하면 돼.
자동차도 마찬가지. 버스 개체가 초록색 마을 버스인지 빨간색 좌석 버스인지 몰라도 돼. 그냥 사람들 태우고 운송하는 버스 역할만 하면 돼.
/ 하나 더. 공연이라고 생각해 보자. '로미오' 역할이 있고 '줄리엣'이라는 역할이 있음.
로미오 역할을 손석구 님이 하든 이민기 님이 하든 상관 없음. 줄리엣 역할을 연습하는 입장에서는 어떤 사람이 로미오 역할을 맡든 내 대사만 연습하면 됨.
손석구 입장에서도 마찬가지. 줄리엣 역할을 누가 할지 몰라. 송혜교 님이 하든 김태희 님이 하든 상관 없어.
누가 오든 어쨌든 나는 이 무대 동선 그대로 움직일 거고, 이 타이밍에 이 대사를 할 거야. 나의 역할은 바뀌지 않아.
내가 무대에서 해야 하는 대사, 내 행동에는 변함이 없어. 줄리엣이라는 역할에 어떤 개체가 오든 나는 영향을 받지 않아.
이게 좋은 객체 지향형 설계임.
/ 어떤 객체가 다른 객체한테 데이터를 요청함. 요청하는 사람을 클라이언트라고 함. 클라이언트 객체는 서버 객체 내부를 몰라도 됨.
서버 객체 내부가 바뀌어도 모름. 심지어 서버 객체 개체가 바뀌어도 나는 모름. 몰라도 됨. 그냥 서버의 역할, 인터페이스만 지켜서 요청하면 됨.
/ 서버 클라이언트가 뭐 대단하게 따로 있는 게 아님. 요청하는 쪽이 클라이언트. 요청 받는 쪽이 서버. 뭐 꼭 페이지를 던져주거나 데이터를 return해야만 서버가 아님.
그냥 요청 받고 그 요청에 해당하는 작업을 수행만 제대로 한다면 서버임.
클라이언트가 서버한테 요청하고, 그 서버가 또 다른 서버한테 요청하는 상황이면 중간에 끼인 서버는 서버이면서 클라이언트.

// 클라이언트가 서버에 요청함. 서버는 역할이 있음.
이 역할을 A개체가 수행하든, B개체가 수행하든, 상관 없음.
구현체를 바꿔 끼워도 다른 곳에 영향도 없고 아무도 모름.
이런 성질이 바로 다형성.
잘 설계한 프로그램은 인터페이스에 변경이 없도록 설계한 프로그램.
그 안에 구현체는 바뀔 수도 있지. 
그러려고 쓰는 게 객체 지향 언어인데.
근데 인터페이스가 흔들리면 그에 엮여 있는 코드들 다 바꾸어야 한다고.
자동차 역할이 비행기 역할로 바껴..? 부품 바 바뀌고 활주로 필요하고 관제탑 필요하고...
연극에서 로미오 역할이 사라지고 잔다르크라는 캐릭터가 새로 생겼어. 대본 다 바뀌고 무대 동선 바뀌고 소품, 의상 다 새로 짜고...

Java를 이용해서 프로그램을 만들 때 이 '다형성'을 살려서 짤 수 있도록 도와주는 도구.
그게 바로 스프링.
